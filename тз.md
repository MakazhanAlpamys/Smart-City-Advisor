Задание для хакатона — «ИИ-помощник по прогулкам в городе Астана»
Кратко:
На входе — набор данных по точкам интереса города (POI) с короткими описаниями. Задача команды — придумать и показать рабочую Retrieval-Augmented Generation (RAG) систему (концепцию + демо), которая в реальном времени подсказывает пользователю куда сходить и что сделать в зависимости от его текущей локации, предпочтений и контекста.

Цель
Создать полезный, безопасный и удобный городской советчик, который сочетает поиск по базе POI и генерацию человекоподобных рекомендаций — чтобы пользователь быстро получил релевантные, конкретные и вдохновляющие подсказки о том, как провести время рядом с собой.

Что ожидается от команд
Концепция продукта — краткое описание идеи, целевой аудитории, сценариев использования.
Демо / прототип — интерактивная демонстрация поведения системы на нескольких сценариях (реальные или смоделированные локации и запросы). Демо может быть как UI-прототипом, так и диалоговой демонстрацией (например, чат).
Описание работы RAG-подхода (высокоуровневое) — как система использует данные POI + генерацию для выдачи рекомендаций (не нужны низкоуровневые архитектурные схемы).
Набор тестовых сценариев — несколько реальных пользовательских кейсов и примеры входных данных + ожидаемый тип ответа.
Краткая презентация (5–7 минут) с бизнес-идей, ограничениями, рисками и планом развития.
Этические и privacy-заметки — как команда учитывает конфиденциальность локации, борьбу с недостоверной информацией и безопасность пользователей.

Данные (что будет в предоставленном датасете POI — пример на человеческом языке)
Название точки (например, «Кафе «Коктем»»).
Координаты (ориентировочно — для сопоставления с локацией пользователя).
Короткое описание.
Категория / теги (кафе, музей, парк, смотровая площадка и т.д.).
Время работы / базовые контактные данные (если есть).
Рейтинг / популярность (если доступно).
Командам не нужно заниматься сбором данных — базовый датасет будет выдан организаторами или предоставлен в виде примеров.

Пользовательские сценарии (примерные)
«Я сейчас возле Mega Silk Way, хочу прогуляться 30–60 минут — что посоветуете?»
«Хочу тихое место с кофе и розеткой недалеко от меня.»
«Еду с ребёнком — где рядом есть детские площадки или музей с интерактивами?»
«Я - турист, у меня 2 часа перед вылетом: что можно посмотреть поблизости?»
«Ищу необычную ночную активность (последние пару часов).»
Для каждого сценария ожидается выдача: 1–3 конкретных пункта с кратким объяснением почему (короткая мотивация), примерным временем на посещение и опцией «что делать там» (конкретное действие/маршрут/рекомендация).

Формат сдачи
Ссылка на репозиторий (код/скрипты — опционально), либо архив с прототипом.
Презентация (PDF/слайды) — 5–7 минут.
Документ с описанием тестовых сценариев и ограничений.

Критерии оценки (в порядке важности)
Релевантность и полезность рекомендаций — насколько ответы соответствуют локации, запросу и ожиданиям пользователя.
Контекстуальность и адаптивность — учёт расстояния, времени, категории пользователя и явных ограничений (время в пути, режим работы).
Качество объяснений — краткое «почему» для каждой рекомендации (помогает доверию).
UX / понятность демо — насколько просто протестировать систему и получить полезный результат.
Креативность и уникальные фишки — специальные фильтры, сценарии, игровые элементы, интеграции (например, микро-маршруты).
Этика и приватность — минимизация хранения точных координат пользователей, прозрачность работы с данными, работа с недостоверной информацией.
Степень завершённости — полнота документации, тестовых сценариев и качества презентации.
Каждый критерий оценивается жюри по шкале (например, 1–10) с суммарным весом по положению организаторов хакатона.

Ограничения и требования (высокоуровневые)
Система должна ясно указывать, какие данные использует (локация пользователя, фильтры, теги).
Избегать уверенных утверждений о доступности места (например, «открыто сейчас»), если данных нет — указывать степень уверенности.
Рекомендации должны быть честно помечены (источник: данные POI vs. сгенерированная идея).

Примеры ожидаемого вывода (кратко, человеческим языком)
Вход: «Я у здании Экспо 2017, хочу что-то необычное на 1–2 часа, пешком.»
Вывод:
Музей будущего искусства (300 м, 5 мин пешком). Почему: интересные интерактивные экспозиции + сейчас идёт вечерняя выставка; план действий: 40–60 мин на экспозицию, затем кофе в подсобном кафе рядом.
Скрытая смотровая - «Лофт на крыше отеля Хилтон» (450 м). Почему: авторская терраса с видом, рекомендовано для фотосессии; план: 20–30 минут, лучше заходить при закрытой погоде.

Рекомендации для команд (опционально)
Сфокусируйтесь на двух–трёх сильных пользовательских сценариях, вместо попытки покрыть всё.
Отдайте приоритет ясности ответов пользователю - короткие, конкретные рекомендации лучше длинных рассуждений.
Пропишите, как система будет обрабатывать сомнительные или неполные данные (например, закрытые места, временные мероприятия).

Стретч-цели (для баллов «креативность»)
Поддержка групповых рекомендаций и планирования (что делать для группы из 3–5 человек).
Генерация мини-маршрута (последовательность 3 мест с маршрутом и временем).
Мультиязычность (советы на языке пользователя).
Интеграция с локальными событиями (временные выставки, вечеринки).
Доступность (рекомендации для людей с ограниченной подвижностью).
